The first version of Script Workflows started from a small product itch: Taras pointed at the trq212 idea of "one-off workflow executions in tasks" and asked what it would look like inside Agent Swarm.

The problem was familiar. Sometimes an agent needs a workflow's power for one job: gather context, call a reusable data script, ask an LLM to compress the result, maybe spawn a reviewer task, then leave an inspectable trail. But requiring a full registered DAG for that kind of job is too much ceremony. The work is real, but it is not always a productized workflow yet.

So we reused the machinery we already trust: scripts, workflow-style steps, task spawning, and the database journal. The result is a one-off TypeScript run that can be launched from an MCP tool, inspected from the dashboard, and resumed from its journal instead of restarting from scratch.

That last sentence is the important part.

## A real loop: launch, journal, inspect

The user-facing path is deliberately boring:

1. Launch with `launch-script-run`.
2. Let the supervisor run it in the background.
3. Inspect with `get-script-run`, `list-script-runs`, or the Script Runs dashboard.

The source is TypeScript:

```ts
export default async function main(args, ctx) {
  const context = await ctx.step.swarmScript("gather-task-context", {
    name: "task-context-gathering",
    scope: "global",
    args: {
      taskId: args.taskId,
      queries: [
        "script workflows durable one-off runs",
        "journal replay label lint",
        "swarm scripts catalog",
      ],
    },
  });

  const brief = await ctx.step.rawLlm("operator-brief", {
    prompt: `Turn this task context into an operator brief:\n${JSON.stringify(context)}`,
  });

  return { context, brief };
}
```

That example is close to how we actually work. Agents do not need to hand-roll multiple task-detail calls, memory searches, deduping, and summarization glue each time. They can call a catalog script like `task-context-gathering`, journal the result, and keep moving.

If the run is interrupted, the script can start again from the top. `gather-task-context` will not fetch again if it already journaled. `operator-brief` will not call the model again if it already journaled. The run resumes by reading its own history.

## Durability is not an implementation detail

Agent work is interruption-prone by default. A useful agent run can span minutes, dozens of tool calls, multiple child tasks, and a human waiting somewhere in Slack. Containers restart. Providers rate-limit. A reviewer asks for a tweak. A process dies after step 7 of 12.

Without durable steps, retry means "do the whole thing again and hope nothing duplicated." That is where agent systems get expensive and weird: duplicate child tasks, repeated outbound messages, repeated LLM calls, and inconsistent partial state.

Script Workflows use the step label as a durability key. Before a step runs, the harness asks the API whether `(runId, label)` already exists in `script_run_journal`.

- If it exists, the harness returns the stored result.
- If it does not exist, the step executes and writes the result or error.

We verified this in live QA after the packaging hotfix. A run called `ctx.step.swarmScript("durable-double", ...)` twice with different source. The second call replayed the first result. The journal had one `durable-double` row, not two.

That sounds small until you apply it to a long swarm job. A run can safely compose "fetch the context", "rank the inputs", "spawn five reviewers", "merge the results", and "write the final report" because each step has a persisted boundary. Retry does not mean duplicate the world.

This is the same reason Agent Swarm itself is AI-Native rather than just AI-First. We are not adding a model call to an old queue and hoping it behaves. We are shaping the execution substrate around the way agents actually fail, resume, delegate, and compound.

## Swarm scripts by default

The other half is the script catalog.

Agent Swarm already had reusable TypeScript scripts callable through `script-run` and from workflow `swarm-script` nodes. Script Workflows make that catalog the default building block for durable one-off jobs.

Every agent gets useful global scripts out of the box:

- `task-context-gathering` gets task details plus deduped memory recall in one call.
- `smart-recall` runs multi-query memory search and reranks the results.
- `compound-insights` builds an all-in-one swarm ops snapshot across tasks, failures, schedules, tool usage, memory health, and per-agent activity.

That is super useful in practice because agents stop re-implementing the same multi-tool chains. They do not need to remember the exact HTTP shape for task details, or hand-roll a memory fan-out every time, or paste the same aggregation code into another prompt. They call the script.

One-off Script Workflow runs combine both pieces:

```ts
const snapshot = await ctx.step.swarmScript("ops-snapshot", {
  name: "compound-insights",
  scope: "global",
  args: { hours: 24 },
});

const risks = await ctx.step.rawLlm("risk-readout", {
  prompt: `Extract the top operational risks:\n${JSON.stringify(snapshot)}`,
});
```

The catalog script does deterministic data work. The workflow journal makes the call resumable. The LLM step is also journaled. If the run dies after the snapshot, the expensive data pull does not repeat. If it dies after the readout, the model call does not repeat.

## Guardrails that matter

Durability needs boundaries. Script Workflows v1 ships with a few practical ones:

- Label lint rejects obvious literal step labels inside loops, because `review` inside a loop is almost always a durability bug.
- `SCRIPT_RUN_MAX_AGENT_TASKS` caps agent-task fan-out. The deployed default is 50; QA confirmed the 51st `agent-task` journal row aborts the run with `aborted_limit`.
- Step and wall-clock caps prevent one-off runs from becoming invisible forever-jobs.
- Terminal runs stay terminal. Cancelling a completed or failed run does not rewrite history.

The terminal states are intentionally plain: `completed`, `failed`, `cancelled`, and `aborted_limit`. `label_lint_violation` is a launch rejection, not a persisted run.

## What this unlocks

For operators, this means fewer "please run these six tools and summarize the result" prompts. You can ask for the job, launch one durable run, and inspect the journal.

For agents, it means less glue. The right move becomes: call a seed script, journal it, use the output, and delegate only when a real agent task is needed.

For the swarm, it is compounding infrastructure. Every useful ad-hoc chain can start as a one-off Script Workflow. If it proves valuable, promote the pieces into catalog scripts or a registered workflow. If it was truly one-time, the run still leaves a clean audit trail.

That is the shape we want: agents doing serious work without turning every serious job into a permanent DAG on day one.
