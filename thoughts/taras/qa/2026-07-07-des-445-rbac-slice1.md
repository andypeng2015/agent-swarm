## 3. Audit behavior (codex-fix surfaces, PR #922 / 356b02d4)

**Trail fidelity.** After Step 1+2 the `permission_audit` table held exactly one
row per `can()` call — 17 rows matching the matrix 1:1 with correct
`principalId`, `verb`, `resourceType`, `decision`, `reason` (rule deny-reasons
verbatim), and `source` (`mcp` vs `http`). Ungated paths (agent-scope
skill/script ops, `task:page:` structural guards) correctly produce no rows.

**3a. Pre-listen wiring.** Restarted the server while a 20ms-interval hammer
fired a gated kv deny. Boot log ordering: `[rbac-audit] Initial retention purge
removed 1 audit row(s)` on **line 7**, `MCP HTTP server running…` on **line 9**
— sink + writer + GC install before `listen()`. The hammer's first-ever
response (HTTP 403 at 18:58:51.45, attempt 236 — i.e. the first request the
socket accepted) is present as a `kv.write.any` deny row. No unaudited startup
window observed.

**3b. 250-call burst.** 250 parallel gated denies completed in **0.59s** wall
(all 403). Control request immediately after: **1.7ms**. Audit delta after ≤3s:
**exactly +250** (threshold flush at 200 + interval flush for the tail). No
blocking, no drops.

**3c. Kill-switch.** Restart with `RBAC_AUDIT_DISABLED=true`: worker global
upsert still 403, lead still 200, kv denies still 403 — **gates enforce,
0 new audit rows** (277 before = 277 after).

**3d. Retention purge.** With the server stopped, inserted a row backdated
`datetime('now','-45 days')`. Next boot: `Initial retention purge removed 1
audit row(s)` and the row is gone (retention default 30d). Verified on **both**
the HTTP server boot and the stdio boot.

**3e. SIGTERM drain (HTTP).** Fired 3 gated denies and SIGTERMed within ~100ms:
count was still 273 (rows buffered, unflushed) right before the signal, **276
after exit** — `shutdown()`'s `stopAuditWriter(); flushAuditBuffer()` drained
the buffer. Clean shutdown log (`MCP HTTP server closed, and database
connection closed`).

**3f. SIGTERM drain (stdio).** `bun src/stdio.ts` over a real stdin pipe:
initialize handshake OK, boot purge removed the backdated row (proves the audit
module is live in the stdio process), SIGTERM → **exit 143** via the explicit
handler (drain sequence identical to 3e). But see Observation B: buffered rows
can't be produced by real stdio traffic today, so the buffered-drain assertion
is only end-to-end-proven on the HTTP server.

## Observations (non-blocking)

**A. `task.fs.mutate` agent branch is unreachable over HTTP with valid auth.**
Every authenticated request is either operator (shared bearer) or user
(`aswt_` token), and both short-circuit to allow; an invalid bearer dies at the
auth layer with 401. So the "non-owner agent → 403" deny case cannot be
constructed over the wire. I diffed against the merge-base: the pre-RBAC
`canMutateTask` had the same order (operator/user return true before agent
checks), so this is faithful parity, not a regression — but it means the fs
deny path is unit-test-only, and the flat-OR rule's agent branch has no live
HTTP consumer until worker fs calls stop using the shared bearer.

**B. No slice-1 gated tool can reach `can()` over pure stdio.** Agent identity
comes exclusively from the `x-agent-id` HTTP header
(`getRequestInfo`, src/tools/utils.ts:28); `StdioServerTransport` carries no
headers, so every gated tool early-returns (verified live: `cancel-task` →
`"Agent ID not found. Set the \"X-Agent-ID\" header."`, no audit row). The
stdio audit wiring (src/stdio.ts:19-58) is therefore defensive/future-proofing:
GC + drain handlers demonstrably run, but the writer can never buffer rows from
real stdio traffic today.
